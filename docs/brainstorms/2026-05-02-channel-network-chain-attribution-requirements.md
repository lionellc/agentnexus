# 渠道链路归因与模型改写检测需求

## Problem Frame

渠道 API 测试台已经能主动测试 OpenAI-compatible 与 Anthropic-compatible 渠道的耗时、首字、usage 和响应体。但用户进一步需要判断：这个渠道是不是被中转、上游更像哪类 provider、中转是否可能做了路由分发，以及是否偷偷改写了用户请求的模型。

首期不追求“穿透中转证明真实上游”。AgentNexus 只基于客户端能观测到的事实做证据归因：展示原始证据、给出置信度、明确不可判断边界，避免把猜测包装成确定结论。

## Requirements

**证据归因口径**

- R1. 系统必须把现有“直连候选/反代候选”升级为链路归因报告，按“已观测事实、强推断、弱推断、不确定、客户端不可判断”分层展示。
- R2. 链路归因报告必须保留并展示本次测试可观测证据：Base URL host、协议、请求模型、响应模型、HTTP 状态、关键响应头、错误体摘要、usage 字段、SSE/首字/完成耗时。
- R3. 系统不得在客户端证据不足时给出确定结论；对最终上游、真实额度来源、真实号池账号这类无法由客户端证明的问题，必须显示“不确定”或“客户端不可判断”。
- R4. 系统必须给出上游候选归因，而不是单个绝对判断；候选可包括官方 API、中转/反代、OpenRouter 类路由、AWS Bedrock 类、Google Vertex 类、未知。

**模型改写检测**

- R5. 系统必须检测“请求模型”和“响应模型字段”是否一致；不一致时标记为“疑似模型被改写”，并在详情中展示请求值、响应值和证据来源。
- R6. 当响应没有模型字段时，系统必须标记为“无法通过响应模型字段判断”，不能默认认为模型未被改写。
- R7. 当响应模型字段与请求模型一致时，系统只能标记为“响应字段一致”，不能证明中转没有改写模型，因为中转可能回填或伪造模型字段。
- R8. 模型改写检测必须进入检查项和链路归因报告，便于用户在表格展开详情中快速看到异常。

**指纹探针**

- R9. 系统必须提供可选的诊断探针模式，用低成本请求收集上游指纹；探针可以覆盖非法模型名、边界参数、stream 事件形态、错误 JSON 形态、usage 字段形态。
- R10. 诊断探针的结果必须以证据表展示，不做模型质量评分，不调用另一个模型进行 AI 评审。
- R11. 探针请求必须复用用户本次输入的协议、模型、Base URL 与 API Key，并沿用现有脱敏规则。

**路由稳定性采样**

- R12. 系统必须支持对同一渠道执行多次小请求采样，用于观察 request id、响应头组合、usage 形态、响应模型、耗时分布是否稳定。
- R13. 如果多次采样出现明显分簇，系统可以标记为“疑似多路由/负载均衡/号池分发”，但必须说明这是行为推断，不是账号池证明。
- R14. 采样结果必须能回到单次测试详情，展示每次采样的核心证据，避免只给汇总结论。

**结果展示**

- R15. 测试结果展开详情必须新增“链路归因”模块，展示上游候选、置信度、证据列表、模型改写检测、不可判断项。
- R16. 链路归因模块必须优先使用表格或分组列表表达证据，避免只有一段解释性文本。
- R17. 归因报告必须能解释为什么 AgentNexus 与中转后台看到的时间、usage 或模型字段可能不一致。
- R18. 所有展示内容必须继续脱敏 API Key、Authorization、x-api-key 等敏感信息。

## Success Criteria

- 用户能在一次渠道测试详情中看到链路归因报告，而不是只有“直连候选/反代候选”标签。
- 用户能看到请求模型与响应模型是否一致；不一致时有明确的“疑似模型被改写”提示。
- 用户能区分“响应字段一致”和“证明未改写模型”是两件事。
- 用户能运行诊断探针，并看到上游候选证据表。
- 用户能通过多次采样观察是否疑似多路由或号池分发，但页面不会把推断说成证明。
- 所有敏感信息在结果表格、展开详情和完整 JSON 中继续脱敏。

## Scope Boundaries

- 首期不接入 New API 或其他中转服务的 trace/log 接口。
- 首期不接入上游账单、额度、官方控制台或第三方后台 API。
- 首期不证明真实账号池、真实额度来源或中转出口网络路径。
- 首期不做定时巡检、告警、导出、趋势图或后台任务队列。
- 首期不做 AI 自动评分，不用另一个模型判断回答质量或上游来源。
- 首期不把 provider 指纹规则做成远程规则市场；规则可先随应用版本维护。

## Key Decisions

- 首期选择“客户端证据优先”：用户已选择不依赖 New API 改造，先把 AgentNexus 自己能看到的证据做扎实。
- 模型改写只做证据检测，不做绝对证明：响应模型字段不一致是强证据；字段一致只是字段一致，不代表中转没有真实改写。
- 路由/号池只做行为推断：多次采样能发现分簇，但没有中转日志时不能证明具体账号池。
- 结果表达必须分层：事实、推断和不可判断分开，避免误导用户。

## Dependencies / Assumptions

- 现有渠道测试台已经记录请求 JSON、响应 JSON、响应头、usage、耗时和完整对话 JSON。
- 现有 OpenAI/Anthropic 解析已经提取响应 `model` 字段；规划阶段需要确认流式和非流式场景的模型字段覆盖一致。
- 现有脱敏逻辑会处理 API Key 和常见鉴权字段；规划阶段需要确认新增归因证据不会绕过脱敏。
