# 模型使用与成本看板 UX 优化 Requirements

## Problem Frame

当前模型使用与成本看板已经具备数据、筛选、图表、定价和请求明细能力，但页面信息架构偏工程功能清单，用户进入页面后需要自己判断：

1. 当前统计是否可信。
2. 成本和请求是否异常。
3. 数据不完整、汇率过期、来源失败分别意味着什么。
4. 下一步应该同步、筛选、查看明细，还是修改定价。

本次优化目标不是重做计费能力，而是在保持既有数据口径的前提下，把页面改成“先判断，再定位，再排障”的用户路径。

## Requirements

**首屏判断区**
- R1. 看板首屏必须优先回答“当前范围内花了多少钱、发生了多少请求、是否存在异常、数据是否可信”。
- R2. 首屏必须集中展示当前时间范围、币种、总成本、请求数、Token、成功/失败/未知状态摘要、数据不完整数。
- R3. 汇率过期、来源覆盖异常、价格来源等解释性状态必须以紧凑状态标识呈现，不应和主指标同权重竞争。
- R4. 首屏主操作必须收敛为用户最常用动作：同步调用、刷新数据；同步价格或定价修改不得抢占主路径。

**任务导向筛选**
- R5. 筛选区必须从“表单网格”改成任务导向结构：时间范围和币种用于定义口径，Agent/模型/状态用于定位异常。
- R6. 状态筛选必须面向用户理解，至少支持全部、成功、失败、未知，并确保 KPI、图表、明细使用同一筛选口径。
- R7. 筛选变更后，页面不得出现“上方指标已变、下方明细仍是旧条件”的口径错位。

**可解释 KPI**
- R8. KPI 必须区分主指标和解释指标：总成本、请求数、Token、成功/失败状态为主指标；不完整记录、汇率状态、可计费请求数为解释指标。
- R9. 不完整记录必须明确表达“缺 model 或 token，不参与成本估算”，避免用户误以为成本已覆盖全部请求。
- R10. 成本 KPI 必须同时解释当前展示币种与 USD/CNY 换算口径；汇率过期时必须显示可理解的过期提示。

**明细排障工作台**
- R11. 页面下半区必须默认服务“定位异常请求”，请求明细应优先于定价规则展示。
- R12. 请求明细必须能快速识别状态、来源、Agent、模型、Input/Output Tokens、成本和是否完整。
- R13. 请求明细空态不能只显示“暂无数据”，必须给出下一步行动建议，例如同步调用、放宽筛选或检查来源状态。
- R14. 定价规则属于低频配置，必须默认折叠或移入次级区域；用户需要修改价格时仍可在看板内完成。
- R15. 如提供模型排行视图，首版必须复用现有模型分布/成本分布数据，不得为了排行单独扩大数据采集范围。

**同步反馈**
- R16. 同步调用进行中必须展示可理解进度，包括当前来源、已处理数量、总数量和失败数量。
- R17. 同步完成后必须展示可关闭的结果反馈，至少包含新增、合并、解析失败或错误信息中的可用字段。
- R18. 同步失败时必须给出可操作提示，不应只暴露内部状态码或原始技术文本。
- R19. 用户关闭完成态或失败态反馈后，不应影响后续刷新和再次同步。

**视觉与交互一致性**
- R20. 本次优化必须延续 AgentNexus 现有设计系统、间距、圆角、颜色和基础组件，不引入新的视觉体系。
- R21. 页面在桌面和窄屏下都必须保持可用：主指标可换行，筛选和操作按钮不能挤压到不可读。
- R22. 交互控件必须保留可访问名称和可测试文本，确保现有自动化测试可覆盖关键行为。

## Success Criteria

- 用户进入看板首屏后，无需滚动即可判断成本、请求规模、异常状态和数据可信度。
- 用户能在同一页面完成“同步数据 -> 判断问题 -> 筛选异常 -> 查看请求明细 -> 必要时修改定价”的闭环。
- 不完整记录和汇率过期不会被误读为已完整计费。
- 定价规则不再打断从概览到明细的主路径。
- 同步完成或失败后的状态提示可关闭，不形成残留 UI 噪音。
- 不新增后端数据源、不改变计费口径、不引入新的图表库或动效依赖。

## Scope Boundaries

- 不改变既有模型使用数据口径：仍然遵循双来源融合、真实数据统计、缺失 model/token 不估算。
- 不新增预算告警、阈值通知、自动优化建议或成本预测。
- 不引入 Recharts、Framer Motion 或新的 UI 框架。
- 不重做全站设计系统，只优化模型看板页面。
- 不把看板拆成多个独立路由，保持在现有 Usage 模块内完成。
- 图表区“成本趋势主图 + 辅助洞察”作为后续优化方向，本首轮不强制纳入。

## Key Decisions

- 新建独立 UX requirements，而不是修改原模型看板 requirements：原文档定义数据口径，本文件定义用户体验改造，避免混淆。
- 首轮优先优化首屏、筛选、KPI、明细和同步反馈：这些问题直接影响用户理解成本，且大多可复用现有数据。
- 定价规则默认降级为次级区域：定价是低频配置，不应阻断高频排障路径。
- 不做图表大重排作为首轮必选：它有价值，但比首屏和明细路径更容易扩大改动面。

## Dependencies / Assumptions

- 既有模型看板数据口径来自 `docs/brainstorms/2026-04-21-model-usage-cost-dashboard-requirements.md`。
- UX 候选方向来自 `docs/ideation/2026-04-29-model-usage-cost-dashboard-ux-ideation.md`。
- 当前前端模型使用数据已包含同步任务进度、来源覆盖、不完整记录、汇率状态、定价行和请求明细字段，可支撑首轮 UX 优化。

## Alternatives Considered

| Alternative | Decision | Rationale |
|---|---|---|
| 把定价规则迁移到独立设置页 | Reject | 会增加跳转成本；默认折叠即可减少干扰。 |
| 首轮加入预算告警和成本预测 | Reject | 超出 UX 优化范围，也违背“不估算缺失数据”的口径。 |
| 首轮强制重排所有图表 | Defer | 有价值，但优先级低于首屏判断和明细排障路径。 |

### Deferred to Planning
